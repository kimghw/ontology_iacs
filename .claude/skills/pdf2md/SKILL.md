# pdf2md — PDF를 구조화 마크다운으로 변환

본 스킬은 PDF 한 파일을 원문 손실 없이 구조화된 마크다운으로 변환하는 절차를 정의한다. 50페이지 단위로 서브에이전트(opus 모델)를 병렬 할당하고, 각 에이전트는 원본을 직접 읽어 담당 구간을 변환하며, 이미지는 별도 파일로 추출하여 링크로 참조한다.

> 범위 밖: 요약·번역·OCR 기반 이미지→텍스트 인식·HTML/DOCX 등 비 PDF 변환. 관련 스킬: `skill-authoring`(본 스킬 작성 규칙).

## 용어

- **오케스트레이터(Orchestrator)**: 본 스킬을 실행하는 주체이자 **사용자의 협력자**. 사용자의 요청을 받아 큐 구성, 서브에이전트 호출·조율, 병합, 이미지 복사·링크 재작성, 검증, 정리, 보고를 담당한다. 본 문서에서 "오케스트레이터"라고 지칭하는 모든 책임은 사용자와 직접 대화하는 이 주체에게 귀속된다.
- **서브에이전트(Sub-agent)**: 오케스트레이터가 Agent 도구로 호출하는 opus 변환 에이전트. 담당 페이지 구간 1개만 변환하며 사용자와 직접 대화하지 않는다. 결과·오류는 오케스트레이터에게만 전달한다.

## 핵심 원칙

- **무손실 보존**: 원문 텍스트는 삭제·의역·요약하지 않는다. 문단·목록·표의 순서를 유지한다.
- **구조화**: 제목 계층(`#`~`######`), 목록, 표, 코드/수식 블록을 마크다운 문법으로 복원한다. 본문 흐름과 무관한 머리말·꼬리말·페이지 번호는 제거한다.
- **첨자·위첨자는 HTML 태그 사용**: 마크다운 표준에는 첨자 문법이 없으므로 아래 첨자(subscript)는 `<sub>...</sub>`, 위 첨자(superscript)는 `<sup>...</sup>`로 표기한다. 예: `R<sub>eH</sub>`, `H<sub>2</sub>O`, `x<sup>2</sup>`, `σ<sub>max</sub>`. 수식 변수의 첨자 의미를 손실 없이 보존하기 위함이며, 이로 인해 markdownlint MD033(no-inline-html)이 위반되면 파일 상단에 `<!-- markdownlint-disable MD033 -->` 디렉티브를 추가한다(태그를 제거하지 않는다).
- **이미지 외부화**: 본문에 포함된 그림·다이어그램은 `assets/` 하위에 저장하고 `![설명](상대경로)` 링크로 참조한다. 원본 캡션이 있으면 유지한다.
- **분할 변환**: 총 페이지 > 50이면 50p 단위로 분할, 구간당 서브에이전트 1개(opus)를 할당한다. 여러 파일을 한 번에 처리할 때도 각 파일을 동일 규칙으로 분해한다.
- **단일 책임 에이전트**: 각 서브에이전트는 담당 페이지 범위만 변환하며 타 구간을 참조·수정하지 않는다.
- **큐 기반 작업 관리**: 작업 단위(= 한 구간)는 `pending → working → done` 디렉토리를 순차 이동하며, 동시에 움직이는 에이전트 수가 병렬 한도를 초과해도 나머지는 큐에서 대기한다.
- **유휴 에이전트 재투입**: 에이전트 1개가 담당 구간을 완료하면 즉시 `pending`에서 다음 작업을 꺼내 진행한다(단, 각 에이전트는 한 번에 1개 작업만 점유).
- **병합 후 단일 산출물**: 파일 1개당 최종 출력은 `.md` 1개. 페이지 단위 구분 마크·구간 간 마크 모두 삽입하지 않는다.
- **검증 필수**: 병합 후 markdownlint로 넘버링·구조 검증을 수행하고 위반 사항을 수정한다.

## 입력 / 출력 규약

- 입력: 단일 또는 다중 PDF 경로 (`<input>.pdf` …).
- **최종 산출 경로**: 최종 `.md` 파일은 **원본 PDF가 위치한 폴더의 이름에 `_md` 접미사를 붙인 폴더**에 저장한다. 예: 원본이 `docs/manual/foo.pdf`이면 최종 산출물은 `docs/manual_md/foo.md`. 동일 폴더 내 다중 PDF는 같은 `<원본폴더>_md/`에 모두 저장.
- 작업 루트: `<workroot>/` (기본값 `./pdf2md_work/`).
- 작업 디렉토리 레이아웃:
  ```
  <workroot>/
  ├── queue/
  │   ├── pending/   ← 분해 직후 대기 중인 구간 작업 파일
  │   ├── working/   ← 현재 에이전트가 점유 중인 작업
  │   └── done/      ← 완료된 작업 (병합 입력)
  └── assets/<input>/  ← 파일별 이미지

  <원본폴더>_md/<input>.md  ← 최종 병합 산출물(원본 폴더 옆에 생성)
  ```
- 큐 작업 파일: `<input>__partNN.task.json` 형태. 내용은 입력 PDF 경로, 페이지 범위, 출력 조각 경로, 이미지 디렉토리, 상태.
- 중간 산출물: `<workroot>/queue/done/<input>__partNN.md` — 병합·검증 완료 후 절차 8에서 `queue/` 전체와 함께 삭제된다.

## 절차

1. **작업 루트 준비**
   - `<workroot>/queue/{pending,working,done}`, `<workroot>/assets/`, `<workroot>/out/` 생성. (`out/`은 병합·린트 스테이징 디렉토리이며 최종 산출 위치가 아니다.)
   - 동시 에이전트 상한(기본 4) 결정. 필요 시 사용자 지시에 맞춰 조정.
2. **사전 조사·이미지 추출** (파일별)
   - `pdfinfo <input>.pdf`로 총 페이지 수 확인.
   - `pdfimages -all <input>.pdf <workroot>/assets/<input>/fig`로 이미지 일괄 추출.
3. **분해 및 큐 적재**
   - 파일당 `ceil(total/50)`개 구간을 계산 → 각 구간을 **작업 파일** `<input>__partNN.task.json`으로 생성.
   - **페이지 범위 분할물 사전 생성(필수)**: 각 구간마다 오케스트레이터가 직접 다음 중 하나의 명령으로 담당 페이지만 포함한 **단일 PDF 추출물**을 미리 생성한다. 추출물은 `<workroot>/queue/pdf_parts/<input>__partNN.pdf` 경로에 저장한다.
     - **권장 (qpdf)**: `qpdf <input>.pdf --pages <input>.pdf <start>-<end> -- <workroot>/queue/pdf_parts/<input>__partNN.pdf` — 다중 페이지를 한 파일로 묶어 출력한다.
     - **대안 (pdfseparate + pdfunite)**: `pdfseparate -f <start> -l <end> <input>.pdf /tmp/_p_%d.pdf && pdfunite /tmp/_p_*.pdf <workroot>/queue/pdf_parts/<input>__partNN.pdf && rm /tmp/_p_*.pdf` — `pdfseparate` 단독 사용은 페이지마다 별도 파일을 만들어 단일 추출물이 되지 않으므로 반드시 `pdfunite`로 묶는다.
     - **절대 금지**: `pdfseparate -f <s> -l <e> ... <out>.pdf` 단독 사용 (페이지별 파일이 생성되어 part_source 규약 위반).
     - **참고(전체가 1파트인 경우)**: 총 페이지가 ≤ 50이고 파트가 1개뿐이면 원본 PDF를 그대로 `cp`하여 `partNN.pdf`로 두어도 무방하다(추출물 생성 의도와 결과가 동일).
     - **텍스트 추출 도구(`pdftotext`, `pymupdf`, `pdfminer` 등) 사용 금지** — 서브에이전트는 이 PDF 추출물을 Read 도구로 직접 읽어 변환한다. 에이전트에게는 이 추출물 경로를 전달하며 **전체 PDF를 재해석시키지 않는다**. 추출물 경로는 task.json에 `part_source` 필드로 기록한다.
   - 모든 작업 파일을 **`queue/pending/`** 에 배치. 분해 직후에는 `working/`·`done/`이 비어 있다.
   - 다중 입력 파일이 있으면 모든 파일의 구간을 하나의 `pending/`에 평탄하게 적재하여 단일 큐로 처리한다.
4. **워커 루프** (유휴 재투입 패턴)
   - 매 라운드에서 `pending/` 최상위 작업을 최대 `min(동시상한, 대기 수)`개 집어 `working/`으로 **원자적 이동**(`mv`)한다. `mv`가 성공한 프로세스만 점유권을 가진다.
   - 이동한 각 작업을 Agent 도구로 opus 서브에이전트 1개에 위임. 단일 메시지 내 다중 Agent 호출로 병렬 기동.
   - 에이전트 완료 후 해당 작업 파일과 생성된 `partNN.md`를 **`done/`** 로 이동한다.
   - 이동이 끝나면 즉시 라운드를 반복 — `pending/`이 빌 때까지 유휴 에이전트는 남은 작업을 이어서 처리한다.
   - 실패한 작업은 `working/` → `pending/`으로 되돌리고 재시도 카운트를 증가. 임계(기본 2회) 초과 시 `failed/`로 분리 후 보고.
5. **에이전트 프롬프트 필수 항목**
   - 담당 PDF 경로, 페이지 범위(예: `pages 51-100`)
   - **담당 페이지 범위 추출물 경로**(`<workroot>/queue/pdf_parts/<input>__partNN.pdf`) — 에이전트는 원본 전체 PDF가 아닌 이 PDF 추출물을 **Read 도구로 직접** 읽어야 한다 (텍스트 추출 도구 사용 금지)
   - 출력 조각 경로(`queue/working/<input>__partNN.md`) 및 최종 이동 시 `done/` 으로 옮긴다는 규약
   - 이미지 디렉토리(상대경로 `../assets/<input>/fig-XXX.ext`)
   - 원문 무손실·구조화·캡션 보존 원칙
   - 페이지 단위 구분 마크·구간 간 마크 모두 삽입 금지
6. **병합** (파일별)
   - 한 입력 파일의 모든 구간이 `done/`에 모인 시점에 해당 파일 병합을 수행한다.
   - `<input>__part01.md`부터 순서대로 단순 연결 → `<workroot>/out/<input>.md` (스테이징 위치). 최종 이동은 검증 통과 후 절차 7에서 수행.
   - 파트 경계에서 잘린 문단·목록·표가 있으면 내용 변경 없이 이어 붙인다(공백 줄만 조정).
   - **6-a. 이미지 동반 이전**: `<workroot>/assets/<input>/` 아래의 모든 이미지 파일을 최종 산출물과 같은 위치의 `<원본폴더>_md/assets/<input>/`로 복사한다. 대상 디렉토리가 없으면 먼저 생성한다. 예: `mkdir -p <원본폴더>_md/assets/<input> && cp <workroot>/assets/<input>/* <원본폴더>_md/assets/<input>/`.
   - **6-b. 이미지 링크 재작성**: 서브에이전트는 `queue/working/` 기준으로 `../assets/<input>/fig-XXX.ext` 형태의 상대경로를 사용했으나, 최종 `.md`는 `<원본폴더>_md/<input>.md`에 위치하므로 해당 경로는 깨진다. 병합된 파일(스테이징·최종 모두)의 이미지 링크를 **최종 위치 기준 상대경로** `assets/<input>/fig-XXX.ext`로 일괄 재작성한다. 간단한 sed 치환 예: `sed -i 's|\.\./assets/<input>/|assets/<input>/|g' <파일경로>` (파일명에 특수문자가 있으면 적절히 이스케이프).
   - **6-c. 책임 주체**: 6-a, 6-b는 **서브에이전트가 아닌 오케스트레이터(본 스킬 실행자)의 책임**이다. 서브에이전트는 `../assets/<input>/...` 규약만 따르고, 이후 이미지 복사·경로 재작성은 병합 단계에서 오케스트레이터가 수행한다.
7. **검증 및 최종 배치**
   - `markdownlint <workroot>/out/<input>.md` 실행(스테이징 파일 대상).
   - 대상 규칙: 제목 계층 순서(MD001), 넘버링 증가(MD029 ordered-list), 중복 제목(MD024), 링크 유효성(MD042), 표 형식 등.
   - **헤딩 순서·표기 검증(필수)**: 원문 PDF의 목차/장·절 번호를 기준으로 최종 `.md`의 헤딩이 다음을 만족하는지 확인한다.
     - **순서 일치**: 원문에 등장한 장·절 순서대로 헤딩이 배열되어 있고 누락·중복·역전이 없다.
     - **계층 일치**: 원문의 장(Chapter)·절(Section)·하위절 깊이에 맞춰 `#`~`######` 레벨이 부여되었고, 한 단계 이상 건너뛰지 않는다(예: `##` 다음 바로 `####` 금지).
     - **표기 일치**: 번호 체계(예: `1`, `1.1`, `1.1.1`) 및 제목 텍스트가 원문과 동일하다(임의 축약·번역·번호 재부여 금지).
     - 위반이 있으면 6-b/7의 자가 수정 절차에 따라 헤딩 레벨·번호·텍스트를 원문 기준으로 교정한 후 재검증한다.
   - **위반 발생 시 자가 수정**: markdownlint가 위반을 보고하면 오케스트레이터가 **직접 해당 `.md`를 수정**하여 위반을 해소한 뒤 재검증한다. 수정 범위는 구조·포맷(제목 계층, 넘버링, 공백, 링크 형식 등)에 한정하며 원문 텍스트·의미는 변경하지 않는다. 원문을 건드려야만 해결되는 위반이거나 자동 판단이 어려운 경우는 수정을 중단하고 사용자에게 보고한다.
   - **사용자 보고**: 재검증까지 완료된 후, 어떤 규칙이 어디서 위반되었고 어떻게 수정했는지(파일·줄·규칙ID·수정 요지)를 **사용자에게 요약 보고**한다. 보고 내용은 `agent_report.md`에도 append한다(타임스탬프 + 작업명 헤더, 프로젝트 CLAUDE.md 규칙).
   - **이미지 링크 해소 검증**: 최종 경로에 배치된 `<원본폴더>_md/<input>.md`에서 모든 이미지 링크(`![...](...)`)를 추출하여 **실제 파일 존재 여부**(`test -f <원본폴더>_md/<링크경로>`)를 확인한다. 하나라도 해소되지 않으면 6-a/6-b를 재수행하고 실패 원인을 보고한다.
     - **이미지 0개 케이스도 명시 보고**: 추출/삽입된 이미지가 0개인 입력 파일이라도 검증을 생략하지 말고 "링크 0/0 통과(이미지 없음)" 형태로 사용자 보고에 명시한다. vacuously true임을 암묵적으로 처리하지 않는다.
   - **린트 통과 후** 스테이징 파일을 최종 경로 `<원본PDF폴더>_md/<input>.md`로 복사한다. 대상 디렉토리(`<원본PDF폴더>_md/`)가 없으면 `mkdir -p`로 생성한다. 동일 원본 폴더의 다중 PDF는 동일한 `<원본폴더>_md/`에 누적 저장된다.
8. **최종 경로 검증 및 정리**
   - **경로 검증**: 각 입력 PDF에 대해 다음을 모두 확인한다.
     - `<원본PDF폴더>_md/<input>.md` 가 존재하고 크기 > 0.
     - `<원본PDF폴더>_md/assets/<input>/` 아래의 이미지 파일 개수가 `<workroot>/assets/<input>/` 원본 개수와 일치.
     - 최종 `.md` 내 모든 이미지 링크가 `test -f` 기준으로 실제 파일을 가리킴.
     - `pending/`·`working/`이 비어 있고, 모든 구간이 `done/`에 존재(실패 항목은 `failed/`로 분리되어 보고됨).
   - **정리(queue 전체 삭제)**: 위 검증이 모두 통과하면 **`<workroot>/queue/` 디렉토리 전체를 삭제한다** (`rm -rf <workroot>/queue`). `pdf_parts/`, `pending/`, `working/`, `done/`, `failed/`, `task.json`, 스테이징 `out/`·중간 `partNN.md`가 모두 이 단계에서 제거된다.
   - `<workroot>/assets/`는 최종 산출 폴더로 이미 복사되었으므로 함께 삭제한다(`rm -rf <workroot>/assets`). 결과적으로 `<workroot>/`는 비거나 완전히 제거되어도 무방하다.
   - **검증 실패 시 절대 삭제하지 않는다.** 실패 원인을 보고하고 큐를 보존한 채 사용자 지시를 기다린다.

## 서브에이전트 지시문 템플릿

```
역할: PDF 구간을 무손실 마크다운으로 변환하는 전문 에이전트.

입력:
- 원본 PDF(참고용): <abs path>
- 담당 페이지 범위: <start>-<end>
- 담당 페이지 추출물(part_source, 이것만 Read 도구로 직접 읽을 것): <workroot>/queue/pdf_parts/<input>__partNN.pdf
- 이미지 디렉토리(이미 추출됨): <workroot>/assets/<input>/
- 출력 파일: <workroot>/queue/working/<input>__partNN.md

규칙:
1. `part_source` PDF 추출물만 **Read 도구로 직접** 읽어 변환한다. `pdftotext`·`pymupdf`·`pdfminer` 등 텍스트 추출 도구/패키지 사용 금지. 원본 전체 PDF나 타 구간도 건드리지 않는다.
2. 원문 텍스트는 삭제·요약·의역 금지. 순서 유지.
3. 제목 계층·목록·표·수식·코드를 마크다운 문법으로 복원.
4. 페이지 번호·반복 머리말/꼬리말 제거.
5. 본문 내 그림은 `![캡션](../assets/<input>/fig-XXX.ext)` 형태의 **queue/working 기준 상대경로**로 삽입(캡션이 있으면 그대로 사용). 이 경로는 병합 단계에서 오케스트레이터가 최종 위치 기준으로 재작성하므로 에이전트는 이 규약만 지키면 된다.
6. 페이지 경계 구분 마크(예: `--- page N ---`)는 삽입하지 않는다.
7. 산출물을 `<workroot>/queue/working/<input>__partNN.md`에 저장한다. `done/`로의 이동은 오케스트레이터가 수행한다. 완료 후 간단 보고(변환 페이지 수, 삽입 이미지 수).
```

## DO

- **서브에이전트는 `part_source` PDF를 Read 도구로 직접 읽어** 시각 구조(제목, 표, 수식, 그림 배치)를 파악하고 마크다운으로 옮긴다.
- 오케스트레이터의 PDF 전처리는 **페이지 분할(`qpdf --pages` 또는 `pdfseparate`+`pdfunite`)과 이미지 추출(`pdfimages`)에 한정**한다. 이 도구들은 변환이 아닌 자료 준비 용도다.
- 매 라운드의 서브에이전트는 **단일 메시지 내 병렬 호출**로 기동한다.
- 큐 이동(`pending → working → done`)은 `mv`로 원자적 처리하여 점유 경합을 막는다.
- 이미지 추출·리네이밍은 구간 분해 전에 **한 번만** 수행하여 구간별 이름 충돌을 피한다.
- 유휴 에이전트는 완료 즉시 `pending/`에서 다음 작업을 이어받는다.
- markdownlint 설정이 없으면 기본값으로 실행하되, 프로젝트에 `.markdownlint.json`이 있으면 그것을 우선한다.
- 표·코드 블록은 언어·정렬 정보를 최대한 보존한다.

## DON'T

- **텍스트 추출 도구·패키지로 변환하지 않는다.** `pdftotext`, `pdfminer`, `pymupdf(fitz)`, `pypdf`, `pdfplumber` 등으로 본문을 뽑아 그대로 마크다운에 붙여넣는 방식 금지 — 제목 계층·표·수식·그림 레이아웃이 깨진다. 서브에이전트는 반드시 Read 도구로 PDF를 직접 읽는다.
- 오케스트레이터가 PDF 본문을 사전 텍스트화하여 서브에이전트에 문자열로 넘겨주지 않는다(서브에이전트의 직접 읽기 원칙을 우회하는 행위).
- 원문 축약·의역·"명확화" 금지.
- 페이지 단위 마크·페이지 경계 마크(`--- page N ---` 등) 중간·최종 산출물 어디에도 삽입하지 않는다.
- 서브에이전트 간 공유 상태·상호 수정 금지.
- 한 에이전트가 `working/`의 두 작업을 동시에 점유 금지(항상 1작업/1에이전트).
- 이미지 인라인 base64 임베드 금지(반드시 파일 링크).
- markdownlint 위반을 무시한 채 완료 선언 금지.

## 체크리스트

- [ ] 입력 파일별 총 페이지 확인, 구간 수 = `ceil(total/50)`
- [ ] `<workroot>/assets/<input>/` 이미지 추출 완료
- [ ] 모든 구간 작업 파일이 `pending/`에 적재됨
- [ ] 라운드마다 opus 서브에이전트 병렬 호출, 동시 상한 준수
- [ ] 각 작업이 `pending → working → done` 순으로 원자 이동

- [ ] 최종 산출물에 페이지 단위·경계 마크 없음

- [ ] 원문 대비 누락 단락·표·그림 없음(스폿 체크)
- [ ] 경로 검증: `<원본폴더>_md/<input>.md` 존재, 이미지 개수 일치, 모든 이미지 링크 해소
- [ ] 검증 통과 후 `<workroot>/queue/` 전체 삭제(`rm -rf`) 및 `<workroot>/assets/` 삭제 — 검증 실패 시 보존

## 자주 등장하는 markdownlint 규칙

PDF→MD 변환 산출물 검증 시 빈번하게 발생하는 위반과 처리 방침. 규칙 ID는 `MD###` 형식이며 슬래시 뒤는 사람이 읽기 쉬운 별칭.

| ID | 별칭 | 의미 | 본 스킬에서의 처리 |
|---|---|---|---|
| MD025 | single-h1 / single-title | 한 문서에 H1은 하나만 | 병합 후 H1이 둘 이상이면 첫 번째를 H1로 유지하고 나머지를 H2로 강등하거나, 의미가 같은 두 H1을 단일 H1로 통합 |
| MD033 | no-inline-html | 인라인 HTML 태그 사용 | 첨자(`<sub>`)·위첨자(`<sup>`)는 원문 의미 보존상 **삭제 금지**. 파일 상단에 `<!-- markdownlint-disable MD033 -->` 디렉티브 추가 |
| MD036 | no-emphasis-as-heading | 강조(`*...*`, `**...**`)를 제목 대용으로 사용 | 개정 이력·문서 종료 표식 등 진짜 제목이 아닌 라인은 emphasis 마커를 제거하고 평문으로 둠. 진짜 제목이면 `#`~`######`로 승격 |
| MD022 | blanks-around-headings | 제목 위/아래에 빈 줄 필요 | 제목 앞뒤에 빈 줄 1줄 삽입 |
| MD031 | blanks-around-fences | 코드 펜스 위/아래에 빈 줄 필요 | 코드 블록 앞뒤에 빈 줄 1줄 삽입 |
| MD040 | fenced-code-language | 코드 펜스에 언어 명시 | 가능하면 언어 지정(`text`, `bash`, `python` 등). 알 수 없으면 `text` |
| MD041 | first-line-h1 | 파일 첫 줄은 H1 | 변환 산출물 첫 줄을 문서 제목 H1로 시작 |
| MD013 | line-length | 한 줄 길이 제한 | 본 프로젝트는 `.markdownlint.json`에서 disable |
| MD029 | ordered-list-prefix | 순서 목록 번호 증가 규칙 | 본 프로젝트는 `.markdownlint.json`에서 disable |

> 처리 원칙: 위반은 **구조·포맷 범위에서만** 자가 수정하며 원문 텍스트·의미는 변경하지 않는다. 자동 판단이 어려우면 수정을 멈추고 사용자에게 보고한다(절차 7 참조).
