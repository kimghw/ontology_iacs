# specification — 최소 작업 단위(Work Unit) 명세 작성 지침

본 스킬은 **에이전트 1개에게 위임 가능한 가장 작은 작업 단위**를 어떻게 명세할지 정의한다. 한 작업 단위는 한 에이전트가 **단독으로 시작 → 수행 → 종료 → 자가검증** 할 수 있어야 한다.

> 팀·프로젝트 수준 계획서가 아니다. 마일스톤·RACI·커뮤니케이션·일정 같은 항목은 본 스킬의 범위가 아니다.

## 에이전트 행동 원칙 (작업 단위 명세에 항상 포함)

모든 작업 단위 문서는 다음 원칙을 명시하거나 암묵적으로 강제해야 한다.

- **예측 금지** — 명세에 없는 입력·동작·결과를 추측하지 않는다. 정해진 입력으로 정해진 절차만 수행한다.
- **모르는 내용 답변 금지** — 확인되지 않은 사실·경로·값·API를 만들어 답하지 않는다. 확인 불가 항목은 Failure·Escalation 경로로 보고한다.
- **불확실하면 멈춘다** — 의심스러우면 진행하지 말고 보고. 부분 진행·임시 추측 후 진행 금지.
- **명세 외 행동 금지** — Steps·Constraints에 없는 부수 작업(파일 정리, 리팩터링, "더 좋은 방법" 적용 등)을 임의로 수행하지 않는다.

---

## 작업 단위 표준 구성

순서대로 포함한다.

1. **ID / Name** — 고유 식별자 (예: `T-042` 또는 의미 있는 슬러그)
2. **Goal** — 한 문장. 결과(outcome) 중심
3. **Inputs** — 필요한 입력(파일·파라미터·전제조건). 경로·형식·존재 여부 명시
4. **Outputs** — 산출물(파일·데이터·값). 경로·형식·개수
5. **Steps** — 순서가 있는 원자적 행동. 에이전트가 수행할 동작 목록
6. **Constraints** — 해서는 안 되는 것, 범위 경계
7. **Acceptance Criteria** — 종료 시 자가검증할 체크 가능한 조건
8. **Failure / Escalation** — 어떤 상황에서 중단하고 어떻게 보고할지

---

## 해야 할 것 (DO)

### 1. 단일 목표
- 한 작업 단위 = 한 Goal = 한 산출물 세트
- "하면서 동시에"가 등장하면 분할 신호

### 2. 입력·출력은 구체적으로
- 입력: 파일 경로(또는 패턴), 형식, 전제 상태
- 출력: 정확한 경로·이름·형식. 와일드카드 금지(개수도 명시)

### 3. 자가검증 가능한 수락기준
- 에이전트가 스스로 통과/실패를 판단할 수 있는 형태
- 가능하면 검증 명령(스크립트·grep·테스트) 명시

### 4. 실패·에스컬레이션 경로 명시
- 어떤 조건에서 작업을 중단하고 사용자에게 보고하는지
- 보고 형식(어디에·무엇을 적을지) 지정

### 5. 원자성
- 작업 도중 중단되어도 외부 상태가 부분적으로 오염되지 않도록
- 가능하면 임시 위치에서 작업 후 최종 위치로 이동

### 6. 멱등성
- 같은 입력으로 두 번 실행해도 같은 결과여야 한다
- 재실행 시 부작용 없도록

### 7. 자기완결성
- 작업 단위 문서만 읽고 에이전트가 시작 가능해야 한다
- 외부 문서 참조는 SSOT 정본 1~2개로 제한

### 8. 길이 제한
- 작업 단위 명세는 **200줄을 넘지 않는다**
- 50줄을 초과하면 단위를 더 작게 분할하거나, 상세 내용을 SSOT 정본으로 옮기고 참조만 남긴다

---

## 하지 말아야 할 것 (DON'T)

- ❌ **여러 목표 묶음 금지** — 두 가지 이상 결과물을 한 단위에 넣지 말 것
- ❌ **모호한 입력/출력 금지** — "관련 파일들", "결과를 정리한 파일" 금지
- ❌ **검증 불가 수락기준 금지** — "잘 처리됨", "정상 동작" 금지
- ❌ **구현 디테일 강제 금지** — 라이브러리 옵션·변수명 같은 에이전트 자율 영역 침범 금지
- ❌ **팀/프로젝트 요소 혼입 금지** — 마일스톤·담당자·일정·커뮤니케이션 채널 금지 (작업 단위 범위 밖)
- ❌ **순환·외부 대기 금지** — 다른 에이전트·사람의 응답을 기다려야 끝나는 작업은 단위가 아니다
- ❌ **부분 성공 허용 금지** — "가능한 만큼만 처리" 금지. 전부 통과 또는 실패 보고
- ❌ **암묵 전제 금지** — "보통은 이렇게 함" 같은 암묵지 금지. 전제는 Inputs에 명시

## 최소 템플릿

```markdown
# T-XXX: <한 줄 작업명>

## Goal
<한 문장 — 결과 중심>

## Inputs
- <경로/패턴> (형식: …, 전제: …)
- <파라미터>: <값 또는 SSOT 키>

## Outputs
- <정확한 경로/이름> (형식: …, 개수: N)

## Steps
1. …
2. …
3. …

## Constraints
- <해서는 안 되는 것>
- <범위 경계>

## Acceptance Criteria
- [ ] <검증 가능 조건 1> (검증: `<명령 또는 방법>`)
- [ ] <검증 가능 조건 2>

## Failure / Escalation
- 조건: <중단 트리거>
- 보고: <어디에·무엇을 적을지>
```

---

## 예시

### 예시 1 — DO

```markdown
# T-017: 입력 디렉토리의 .md 파일 헤딩 추출

## Goal
<INPUT_DIR>의 모든 .md 파일에서 헤딩 구조를 추출하여 단일 TSV로 산출한다.

## Inputs
- <INPUT_DIR>/*.md (형식: UTF-8 마크다운, 전제: 파일 ≥ 1개 존재)
- max_level: 6

## Outputs
- <OUTPUT_PATH>/headings.tsv (형식: TSV, 컬럼: file, level, line, text)

## Steps
1. <INPUT_DIR>의 .md 파일 목록을 정렬된 순서로 수집
2. 각 파일에서 ATX/Setext 헤딩 라인을 추출 (level ≤ max_level)
3. 행을 file 오름차순, line 오름차순으로 정렬해 단일 TSV로 기록

## Constraints
- 코드 블록(```) 내부의 # 라인은 헤딩으로 보지 않는다
- 입력 파일을 수정하지 않는다

## Acceptance Criteria
- [ ] <OUTPUT_PATH>/headings.tsv 존재 (검증: `test -s <OUTPUT_PATH>/headings.tsv`)
- [ ] 첫 행이 헤더 `file\tlevel\tline\ttext`
- [ ] level 컬럼 값이 모두 1~max_level 범위

## Failure / Escalation
- 조건: 입력 파일 0개, 또는 단일 파일에서 헤딩 0개 검출
- 보고: `report.md`에 파일명·사유 append 후 종료
```

### 예시 1' — DON'T (같은 작업)

```markdown
# 헤딩 작업

마크다운 파일들에서 헤딩을 적절히 뽑아 정리해줘.   ← Goal·Inputs·Outputs·검증 모두 모호
결과는 파일로 저장.   ← 경로·형식·개수 없음
가능한 만큼 처리하고 안 되면 알려줘.   ← 부분 성공 허용·검증 불가
```

## 작성 후 체크리스트

- [ ] Goal이 한 문장이고 결과 중심
- [ ] Inputs에 경로·형식·전제 모두 명시
- [ ] Outputs에 정확한 경로·형식·개수 명시 (와일드카드 없음)
- [ ] 모든 Acceptance Criteria가 자가검증 가능
- [ ] Failure 조건과 보고 위치·형식 명시
- [ ] 모호 표현("적절히", "정상", "가능한 만큼") 0건
- [ ] 한 단위에 목표 1개·산출물 세트 1개
- [ ] 외부 사람·에이전트 응답 대기 없음
- [ ] 재실행 시 멱등 (같은 입력 → 같은 결과)
- [ ] 본 문서만으로 에이전트가 시작 가능
- [ ] 전체 명세 길이 ≤ 200줄
- [ ] 에이전트 행동 원칙(예측 금지·모르는 내용 답변 금지·불확실 시 중단·명세 외 행동 금지)이 본문 또는 Constraints에 포함되어 있음

---

## 에이전트에게 항상 상기시킬 것 (Reminder)

작업 단위를 위임받은 에이전트는 작업 시작 전·종료 전 다음을 자기 점검해야 한다.

- 나는 **명세에 적힌 것만** 수행하는가? 없는 동작을 추측해 추가하지 않았는가?
- 내가 답하거나 산출한 값은 **검증된 사실**인가? 모르는 것을 그럴듯하게 만들어내지 않았는가?
- 불확실한 부분이 있으면 **멈추고 보고**했는가? 임시로 메우고 진행하지 않았는가?
- 명세에 없는 "친절한 추가 작업"을 자발적으로 수행하지 않았는가?

> 위 4가지 중 하나라도 어긋나면 그 작업 단위는 실패로 간주하고 Failure·Escalation 경로로 보고한다.
