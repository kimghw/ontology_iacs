# skill-authoring — Claude Code Skill 작성 지침

본 스킬은 **재사용 가능한 절차·노하우 패키지**인 Claude Code Skill을 어떻게 작성할지 정의한다. 한 스킬은 Claude가 **트리거 조건을 보고 자동 또는 명시적으로 호출**하여 사용할 수 있어야 한다.

> 일회성 작업 지시(작업명세서)가 아니다. 영속적 자산으로서 여러 작업에서 재사용되는 **HOW**를 기술한다. 작업명세서(WHAT)는 `specification` 스킬 참조.

---

## 0. 스킬 vs 인접 개념

| 개념 | 본질 | 호출 | 수명 |
|:---|:---|:---|:---|
| **Skill** | 재사용 절차/노하우 (HOW) | 자동(설명 매칭) 또는 `/name` | 영속 |
| **Subagent** | 격리 컨텍스트에서 실행되는 별도 에이전트 | `Agent` tool 위임 | 호출 단위 |
| **Slash Command** | 레거시 수동 명령(스킬이 대체) | `/cmd` | 영속 |
| **작업명세서** | 일회성 작업 지시(WHAT) | 사람/오케스트레이터 | 작업 종료 시 |

## 1. 파일 위치 및 구조

```text
<scope>/skills/<skill-name>/
├── SKILL.md              ← 필수 엔트리포인트
├── reference.md          ← (선택) 긴 참조 자료
├── examples.md           ← (선택) 사용 예제
├── scripts/helper.py     ← (선택) 실행 스크립트
└── assets/...            ← (선택) 템플릿·샘플 입력
```

**스코프 우선순위** (Enterprise > Personal > Project):

| 스코프 | 경로 | 사용 시점 |
|:---|:---|:---|
| Personal | `~/.claude/skills/<name>/SKILL.md` | 모든 프로젝트에서 재사용 |
| Project | `.claude/skills/<name>/SKILL.md` | 이 저장소 전용 |
| Plugin | `<plugin>/skills/<name>/SKILL.md` | 플러그인 배포 |

- 디렉토리명 = 스킬명 = `name` frontmatter 값 (소문자·하이픈, ≤64자).
- `SKILL.md`가 유일한 필수 파일. 나머지는 본문에서 **참조 링크**로 지연 로드.

## 2. YAML Frontmatter

```yaml
---
name: <skill-name>
description: <한 문단 — 무엇을·언제·언제 쓰지 말지. TRIGGER/DO NOT TRIGGER 포함. 250자 이내 권장>
---
```

### 2.1 필수·권장 필드

| 필드 | 필수 | 설명 |
|:---|:---:|:---|
| `name` | 권장 | 디렉토리명과 일치. `/name`으로 수동 호출 가능 |
| `description` | **필수** | Claude가 **항상 컨텍스트에 로드**하는 유일한 부분. 여기서 트리거 판단 |

### 2.2 선택 필드

| 필드 | 용도 |
|:---|:---|
| `disable-model-invocation: true` | 사용자만 호출 가능 (부작용 있는 작업에 권장). description이 컨텍스트에서 완전히 제거됨 |
| `user-invocable: false` | Claude만 참조 가능 (배경지식용) |
| `allowed-tools: Read Grep` | 도구 **사전 승인** (제한이 아님 — 도구 차단은 permission deny 규칙으로) |
| `context: fork` | 격리 서브에이전트로 실행 |
| `argument-hint: [issue-number]` | `/name` 자동완성 시 표시되는 인자 힌트 |
| `model: opus` | 스킬 실행 시 모델 오버라이드 (`opus`, `sonnet`, `haiku`) |
| `effort: high` | 스킬 실행 시 사고 수준 오버라이드 (`low`, `medium`, `high`) |
| `agent: Explore` | 서브에이전트 타입 지정 (`Explore`, `Plan`, `general-purpose`, 또는 `.claude/agents/` 커스텀) |
| `paths: ["src/**/*.ts"]` | 자동 활성화를 특정 파일 패턴으로 제한 (YAML 리스트) |
| `hooks: ...` | 스킬 스코프 라이프사이클 훅 |
| `shell: bash` | 셸 지정 (`bash` 기본, `powershell` 가능) |

### 2.3 문자열 치환 변수

스킬 본문에서 사용 가능한 런타임 변수:

| 변수 | 설명 |
|:---|:---|
| `$ARGUMENTS` | `/name arg1 arg2` 호출 시 전체 인자 문자열 |
| `$1`, `$2`, … | 위치별 인자 (셸 스타일) |
| `${CLAUDE_SKILL_DIR}` | 스킬 자신의 디렉토리 절대경로 |
| `${CLAUDE_SESSION_ID}` | 현재 세션 ID |

### 2.4 description 작성 규칙

- **구조**: `<무엇인지 한 줄> + <언제 사용 — 구체 상황> + TRIGGER when … + DO NOT TRIGGER when …`
- **250자 내**. 초과 시 잘릴 수 있음. (description 예산: 컨텍스트 윈도우의 1%, 항목당 최대 250자)
- **진술형** ("…할 때 사용") 권장. 명령형("…해라")은 해석 경직.
- **의도 매칭으로 작성**: "이 스킬이 무엇을 하는지"(기능 목록)가 아니라 **"사용자가 무엇을 원할 때"**(의도)로 TRIGGER를 쓴다. 라우터가 사용자 의도와 매칭하므로 의도 표현이 더 정확하게 트리거된다.
- **핵심 키워드**를 앞쪽에 배치 (PDF 변환, 커밋 리뷰, 헤딩 추출 등).
- 유사 스킬과 **경계**를 명시 — `specification` 스킬 두 개가 공존하면 DO NOT TRIGGER 절로 구분 필수.

**좋은 예**:

```yaml
description: PDF를 마크다운으로 변환. 원본 구조·페이지 경계 보존. 50p 단위 멀티에이전트 분할 변환 후 병합. TRIGGER when 사용자가 PDF→MD 변환·정규화·청킹 전처리 요청. DO NOT TRIGGER when HTML/XML/JSON 변환, OCR 이미지 추출.
```

**나쁜 예**:

```yaml
description: 파일을 변환하는 유용한 스킬입니다. 필요할 때 사용하세요.
```

→ 무엇을·언제·경계 모두 부재. 트리거 못 탐.

## 3. 본문 구조

### 3.1 권장 섹션 (순서)

1. **제목 + 한 줄 요약** — `# <name> — <역할>`
2. **범위/경계** — 무엇을 다루고 무엇을 안 다루는지 (blockquote)
3. **핵심 원칙** — 이 절차가 지키는 불변 규칙
4. **사용 도구** — 스킬이 사용하는 도구 목록과 용도 (Read, Grep, Bash, Agent 등)
5. **표준 구성 / 템플릿** — 재사용할 구조·스키마·파일 형식
6. **DO / DON'T** — 구체적 가이드
7. **예시** — 좋은 예 + 나쁜 예 대비
8. **체크리스트** — 작업 완료 전 자가검증 항목
9. **참조** — 보조 파일·SSOT 정본 링크

### 3.2 길이

- **500줄 이내 권장**. 초과 시 상세 내용을 `reference.md`·`examples.md`로 분리.
- Progressive disclosure: 본문은 스킬 **호출 시점에만** 로드되므로 필수 지시만 남기고 나머지는 링크.

### 3.3 말투

- **진술형·지시형** 혼용. "…은 …이다" / "…한다" / "…하지 않는다".
- 2인칭("너는") 금지. 에이전트 중립적 표현.
- 이모지·장식 금지.

## 4. 보조 파일 운영

| 파일 | 용도 | 로드 시점 |
|:---|:---|:---|
| `reference.md` | 긴 스키마·API·용어집 | 본문에서 필요 시 Read |
| `examples.md` | 실제 입출력 예시 모음 | 본문에서 필요 시 Read |
| `scripts/*.py` | 스킬이 실행할 유틸리티 | 스킬이 Bash로 실행 |
| `.venv/` | 스킬 전용 Python 가상환경 (gitignore 대상). 패키지 설치·스크립트 실행 시 `${CLAUDE_SKILL_DIR}/.venv/bin/pip`, `…/python` 사용 | 스킬이 Bash로 생성·사용 |
| `assets/*` | 템플릿·샘플 파일 | 스킬이 복사·참조 |

- 보조 파일은 **SKILL.md 본문에서 상대경로로 명시적으로 참조**해야 Claude가 찾는다.
- 용량이 큰 참조 자료(수천 줄)는 무조건 별도 파일로.

## 5. Progressive Disclosure — 왜 description이 중요한가

Claude의 컨텍스트에 **항상 실리는 것은 `description`뿐**이다. 본문 SKILL.md는 스킬이 호출된 순간에만 로드된다.

따라서:

- `description`은 **"언제 이 스킬을 쓸지" 판단에 필요한 모든 신호**를 담아야 한다.
- 본문은 **트리거 이후의 세부 지침**만 담는다. 트리거 판단용 정보를 본문에만 숨기면 영원히 호출되지 않는다.
- 유사 스킬이 많을수록 description의 **경계 명시**(DO NOT TRIGGER)가 중요.

### 5.1 컨텍스트 예산과 수명

- **description 예산**: 컨텍스트 윈도우의 약 2% (폴백 16,000자). 항목당 최대 250자. `SLASH_COMMAND_TOOL_CHAR_BUDGET` 환경변수로 오버라이드 가능.
- **본문 수명**: 렌더링된 SKILL.md는 대화 메시지로 진입, 자동 압축 시 최신 호출 1건이 유지된다. 재호출하면 이전 사본을 대체한다.

### 5.2 지식과 지시의 분리

- **참조 자료**(스키마·용어집·API 명세)는 보조 파일에 둔다 — 조회 대상.
- **행동 규칙**(절차·DO/DON'T·체크리스트)은 SKILL.md 본문에 둔다 — 실행 지침.
- 이 구분이 모호하면 "Claude가 따르는 것인가(지시), 찾아보는 것인가(지식)"로 판단한다.

### 5.3 핵심 규칙 이중 배치 (샌드위치 패턴)

긴 본문(300줄+)에서 핵심 불변 규칙은 **상단(핵심 원칙)과 하단(체크리스트/DON'T) 양쪽에 배치**한다. 긴 컨텍스트 중간에 있는 규칙은 준수율이 떨어진다. 상단에서 선언하고 하단에서 재확인하면 준수율이 높아진다.

## 6. DO / DON'T

### DO

- 디렉토리명·`name`·파일 내 제목을 **동일한 슬러그**로 맞춘다.
- description에 **TRIGGER / DO NOT TRIGGER**를 명시한다.
- 본문은 **재사용 가능한 절차·원칙·템플릿**만 담는다.
- 긴 참조·예시는 보조 파일로 **지연 로드**한다.
- 부작용 있는 스킬(커밋·배포·외부 API 호출)은 `disable-model-invocation: true`를 고려한다.
- SSOT 정본이 있으면 그곳을 참조하고 스킬에 중복 정의하지 않는다.
- 스킬이 서브에이전트를 호출하는 경우, 본문을 **오케스트레이터 섹션과 서브에이전트 지시문 템플릿으로 물리적으로 분리**하고 템플릿은 호출 시 문자 그대로 복사한다(아래 §13 참조).
- 스킬이 서브에이전트를 호출하는 경우, **산출물은 파일로 기록하고 반환은 요약·이슈만** 받도록 프롬프트·출력 계약에 명시한다(아래 §13 참조).

### DON'T

- ❌ **일회성 작업 내용**을 스킬에 담지 않는다 (그건 작업명세서).
- ❌ **250자 넘는 description** 금지 — 잘림 위험.
- ❌ **모호한 트리거** ("필요할 때 사용") 금지.
- ❌ **중복 스킬명** 금지 — 기존 `specification`처럼 같은 이름이 두 개면 Claude가 어느 것을 호출할지 불명확.
- ❌ **500줄 초과 본문** 금지 — 분리.
- ❌ **특정 프로젝트 경로 하드코딩** 금지 (Personal 스킬의 경우). 변수·패턴으로 추상화.
- ❌ **실행 결과 예측·가정** 금지. 스킬은 절차만 담고 판단은 호출 시점 Claude에 위임.

## 7. 최소 템플릿

```markdown
---
name: <skill-slug>
description: <한 줄 요약>. <언제 쓰는지 — 구체 조건>. TRIGGER when <조건>. DO NOT TRIGGER when <경계>.
---

# <skill-slug> — <한 줄 역할>

본 스킬은 <무엇을> 위해 <어떤 원칙으로> 수행하는 절차를 정의한다.

> 범위 밖: <다루지 않는 것>. 관련 스킬: <다른 스킬 이름>.

## 핵심 원칙
- <불변 규칙 1>
- <불변 규칙 2>

## 사용 도구
| 도구 | 용도 |
|:---|:---|
| `Read` | <용도> |
| `Bash` | <용도> |

## 절차
1. <단계 1>
2. <단계 2>
3. <단계 3>

## 입력 / 출력 규약
- 입력: <형식·경로 패턴>
- 출력: <형식·경로 패턴>

## DO
- <권장 행동>

## DON'T
- ❌ <금지 행동>

## 예시
### 좋은 예
<코드/텍스트>

### 나쁜 예
<코드/텍스트>

## 체크리스트
- [ ] <자가검증 항목 1>
- [ ] <자가검증 항목 2>

## 참조
- [reference.md](reference.md) — 상세 스키마
- <SSOT 정본 경로>
```

## 8. 작성 후 체크리스트

- [ ] 디렉토리명 = `name` frontmatter = 본문 제목 슬러그
- [ ] `description` ≤ 250자
- [ ] `description`에 TRIGGER / DO NOT TRIGGER 명시
- [ ] 유사·중복 스킬과 경계 구분
- [ ] 본문 ≤ 500줄 (초과 시 보조 파일 분리)
- [ ] 본문이 재사용 가능한 절차·원칙만 담음 (일회성 작업 내용 없음)
- [ ] 보조 파일은 본문에서 상대경로로 명시 참조
- [ ] 부작용 있는 작업이면 `disable-model-invocation` 고려
- [ ] SSOT 정본 중복 정의 없음
- [ ] 장식·이모지·2인칭 없음
- [ ] 서브에이전트 호출 스킬이면 오케스트레이터/서브에이전트 본문이 분리됨(§12), 반환 규약 명시(§13)

## 9. 흔한 함정

| 증상 | 원인 | 해결 |
|:---|:---|:---|
| 스킬이 호출되지 않음 | description이 모호하거나 트리거 키워드 부재 | TRIGGER when … 명시, 도메인 키워드 앞쪽 배치 |
| 잘못된 스킬이 호출됨 | 유사 스킬과 경계 미분리 | DO NOT TRIGGER when … 추가 |
| description 잘림 | 250자 초과 | 핵심 키워드를 앞쪽에 |
| 본문이 너무 길어 로드 부담 | 참조·예시를 본문에 인라인 | `reference.md`·`examples.md` 분리 |
| 스킬이 부작용을 내고도 경고 없이 실행 | `disable-model-invocation` 미설정 | 사용자 확인 필요한 스킬에 설정 |
| 중복 스킬명 | 동일 `name`을 가진 스킬 2개 공존 | 슬러그 명확화(`specification-workplan` vs `specification-workunit`) |
| 서브에이전트가 스킬의 핵심 규칙을 안 지킴 | 오케스트레이터가 템플릿을 요약·재작성하여 호출 — 서브에이전트는 SKILL.md를 보지 못함 | §12 분리 구조로 작성 + "문자 그대로 복사" 명시 |
| 서브에이전트 결과가 부모 컨텍스트를 가득 채움 | 본문을 메시지로 반환 | §13 반환 규약 — 산출물은 파일, 반환은 요약·이슈만 |
| `allowed-tools`로 도구를 차단했다고 착각 | `allowed-tools`는 사전 승인일 뿐 제한이 아님 | 차단은 permission deny 규칙으로. managed policy `ask` 규칙은 `allowed-tools`로 우회 불가 |
| 정보가 부족한데 추측으로 진행 | 에이전트가 도구로 확인하지 않고 가정 | 스킬 본문에 "도구로 확인. 추측 금지" 에이전트 리마인더 포함 |

## 9.1 에이전트 리마인더 (Agentic Reminders)

스킬 본문이 길거나 복잡한 경우, 다음 세 가지 리마인더를 본문 상단 또는 핵심 원칙에 포함하면 준수율이 높아진다:

- **끈기**: "작업이 완전히 해결될 때까지 계속 진행한다. 중간에 제어를 넘기지 않는다."
- **도구 우선**: "정보가 필요하면 도구로 확인한다. 추측하지 않는다."
- **계획 후 실행**: "각 단계 전에 계획을 세우고, 결과를 반영한다."

---

## 10. 평가 주도 개발 (Evaluation-Driven Development)

스킬을 작성하기 전에 **테스트 시나리오 3건 이상**을 먼저 정의한다.

1. 스킬 없이 기본 Claude로 시나리오를 실행 → 베이스라인 측정
2. 스킬 초안을 작성하고 동일 시나리오 실행 → 개선 여부 확인
3. 반복 개선: 실패 시나리오를 분석하여 본문·description 수정

**Claude A / Claude B 패턴**: 한 Claude 인스턴스(A)가 스킬을 작성·수정하고, 다른 인스턴스(B)가 스킬을 테스트한다. B의 행동을 관찰하여 A가 개선하는 루프.

## 11. 항상 상기할 것

- 스킬은 **언제 쓸지(description)**가 본문만큼 중요하다.
- 스킬은 **재사용 자산**이다. 일회성 지시는 작업명세서로.
- 스킬은 **지연 로드**된다. description만 항상 메모리에 있다.
- 유사 스킬이 많을수록 **경계 명시**가 품질을 결정한다.
- 서브에이전트를 호출하는 스킬은 **두 주체로 본문이 분리**되어야 한다 — 서브에이전트는 SKILL.md를 보지 못한다.
- Claude는 이미 유능하다. **Claude가 모르는 맥락만** 추가한다. 당연한 것을 장황하게 설명하지 않는다.

## 12. 오케스트레이터 / 서브에이전트 본문 분리

스킬이 내부에서 서브에이전트(Agent 도구)를 호출하는 경우, **SKILL.md 본문은 두 주체의 책임을 구조적으로 분리**하여 작성한다. 서브에이전트는 스킬 파일을 읽지 않고 오케스트레이터가 보낸 프롬프트 텍스트만 본다. 따라서 "스킬 본문에는 있지만 프롬프트에는 빠진 규칙"은 곧 품질 저하 지점이 된다. 본 저장소의 [pdf2md/SKILL.md](../pdf2md/SKILL.md)가 이 패턴의 정본 구현이다(3절 오케스트레이터 / 4절 서브에이전트 지시문 템플릿).

### 12.1 분리 원칙

- **오케스트레이터 섹션**: 스킬을 실행하는 주체(사용자와 대화하는 Claude)가 수행하는 작업. 입력 준비·분해·큐 관리·서브에이전트 호출 조립·병합·검증·정리·사용자 보고.
- **서브에이전트 지시문 템플릿 섹션**: Agent 도구로 호출되는 에이전트에 **문자 그대로 복사되는 프롬프트 페이로드**. 이 섹션 전체가 통째로 프롬프트가 되며, 에이전트는 SKILL.md의 다른 섹션을 읽지 않는다 → **자체완결**되어야 한다.
- **참조가 아닌 복제**: 핵심 원칙·DO/DON'T·체크리스트가 양쪽 모두에 필요하면 두 섹션에 **각각 복제**한다. "위 섹션 참조"라고 쓰면 서브에이전트는 그 위 섹션을 보지 못한다.

### 12.2 권장 골격

```markdown
## 문서 구조 (안내)
본 스킬은 오케스트레이터와 서브에이전트 두 주체의 책임을 분리한다.
- N절: 오케스트레이터 작업 범위 — 스킬 실행자 자신이 수행
- M절: 서브에이전트 지시문 템플릿 — Agent 호출 프롬프트로 문자 그대로 복사

## N. 오케스트레이터 작업 범위
### N.1 핵심 원칙 (오케스트레이터)
### N.2 절차
### N.3 조건 플래그·메타데이터 확정 규칙   ← 분해 시점에 결정되어 프롬프트 슬롯에 주입되는 값
### N.4 서브에이전트 프롬프트 조립 규칙     ← "M절을 문자 그대로 복사하고 슬롯만 치환"
### N.5 DO / DON'T (오케스트레이터)
### N.6 체크리스트 (오케스트레이터)

## M. 서브에이전트 지시문 템플릿
> 오케스트레이터 주의: 이 섹션 전체를 Agent 호출 프롬프트에 문자 그대로 복사한다.
> 슬롯(경로·플래그)만 치환하며, 그 외는 축약·생략·재작성 금지.

### M.1 역할 / 입력 (경로 플레이스홀더)
### M.2 작업 조건 (조건 플래그 슬롯)
### M.3 핵심 원칙 (서브에이전트가 지킬 불변 규칙)
### M.4 절차
### M.5 불변 작업 규칙
### M.6 조건부 분기 규칙 (M.2 플래그에 의한 분기)
### M.7 도메인별 주의사항 (예: lint 사전 회피)
### M.8 DO / DON'T (서브에이전트)
### M.9 자가 체크리스트 (보고 전 점검)
### M.10 완료 보고 형식 (오케스트레이터에 돌려줄 구조화 보고)
```

### 12.3 슬롯과 사전 플래그

서브에이전트 템플릿 안에는 **호출 시점에만 결정되는 변수**(경로, 범위, 인덱스, 부울 플래그 등)를 명시 슬롯으로 둔다. 슬롯 값은 오케스트레이터가 N.3 규칙에 따라 분해/사전조사 단계에서 산출하여 N.4 조립 단계에서 주입한다.

- **사전 결정 가능한 값**(인덱스, 총 개수, 첫/마지막 여부, 리소스 카운트 등)은 슬롯으로 사전 주입하여 서브에이전트가 추측·재계산하지 않게 한다.
- **사전 결정 불가한 값**(원문에서 발견되어야 하는 특수 표기 여부 등)은 슬롯으로 두지 말고 서브에이전트가 직접 관찰하여 **완료 보고로 역보고**하게 한다. 오케스트레이터는 보고를 받아 병합·후처리에 반영한다.

### 12.4 "문자 그대로 복사" 규칙이 필요한 이유

- 서브에이전트는 SKILL.md를 읽지 않는다. 오케스트레이터가 요약하면 핵심 금지 규칙·도구 제약·조건부 분기·실패 방지 노하우가 누락된다.
- 누락된 규칙은 보통 검증 단계에서야 발견되며, 자가 수정 부담과 사용자 보고 잡음을 키운다.
- 따라서 서브에이전트 섹션은 **자체완결**이어야 하고, 오케스트레이터의 조립 절차(N.4)에 "축약·생략·재작성 금지" 문구를 명시한다.

### 12.5 DO / DON'T

#### DO (멀티에이전트)

- 서브에이전트가 보는 섹션과 오케스트레이터가 보는 섹션을 **물리적으로 분리**한다(별도 절·번호 부여).
- 서브에이전트 섹션 첫 머리에 "이 섹션 전체를 프롬프트로 문자 그대로 복사" 안내(blockquote)를 둔다.
- 슬롯 표기(`<workroot>`, `<input>`, `<part_index>` 등)를 일관되게 사용하고, 슬롯 목록을 N.3 / M.2에 표로 둔다.
- 같은 규칙이 양쪽에 필요하면 **간결히 복제**한다.
- 사전 결정 가능한 값은 슬롯으로, 관찰이 필요한 값은 완료 보고 역방향으로 명확히 갈라둔다.

#### DON'T (멀티에이전트)

- ❌ 서브에이전트 섹션에서 오케스트레이터 섹션을 "위 N절 참조"로 가리키지 않는다 — 서브에이전트는 그 섹션을 보지 못한다.
- ❌ 오케스트레이터 섹션에서 "에이전트가 알아서 판단"하도록 모호하게 두지 않는다. 결정 책임을 양쪽에 명시한다.
- ❌ 슬롯이 아닌 자리에 동적 값을 인라인으로 기록하지 않는다(템플릿이 깨진다).
- ❌ 서브에이전트 템플릿을 호출마다 즉흥 재작성하지 않는다(드리프트 발생).

## 13. 서브에이전트 결과 반환 규약

스킬이 내부에서 서브에이전트를 호출하는 경우, **서브에이전트가 작업 본문을 호출자에게 통째로 반환하게 두면 안 된다**. 본문이 부모 컨텍스트를 채워 progressive disclosure의 이점이 사라지고, 여러 에이전트의 결과가 합쳐지면 토큰이 폭증한다.

### 13.1 원칙

- **본문은 파일로**: 변환된 마크다운·생성된 코드·리팩터링 결과 등 "문서에 써야 할 산출물"은 에이전트가 **직접 파일로 기록**한다. 경로 규약은 스킬 본문에서 미리 정의한다.
- **반환은 요약·이슈만**: 에이전트가 부모로 돌려보내는 텍스트는 ①작업 완료 여부 ②산출물 경로 ③다음 단계 결정에 필요한 이슈·모호점·판단 근거만 담는다.
- **결정이 필요한 항목만 에스컬레이션**: 부모가 보고 조치해야 할 것만 반환한다. "무엇을 했는지"는 파일과 git diff가 말해준다.
- **조용한 성공**: 이슈가 없으면 한두 줄("완료. 산출물: `path`. 이슈 없음.")로 끝낸다.

### 13.2 표준 반환 포맷 (권장)

```yaml
status: ok | needs_review | failed
outputs:
  - <파일 경로 1>
  - <파일 경로 2>
issues:        # 없으면 빈 배열
  - severity: high|medium|low
    where: <경로/라인/범위>
    note: <한 줄 설명>
notes: <옵션, 부모가 판단할 수 있는 짧은 맥락>
```

### 13.3 스킬 본문에서 해야 할 일

- 서브에이전트 프롬프트 템플릿에 **"산출물은 <경로>에 쓰고, 반환은 요약·이슈만"** 문장을 고정으로 포함한다.
- 산출물 경로·네이밍 규약을 스킬이 선제적으로 정의한다(에이전트가 경로를 창작하지 않도록).
- 반환 포맷을 본문에 명시한다(위 §13.2).

### 13.4 DON'T

- ❌ 서브에이전트가 변환 결과 전체를 메시지로 덤프하게 두지 않는다.
- ❌ "작업 내용을 자세히 보고하라"는 모호한 지시를 프롬프트에 남기지 않는다 — 보고 범위를 이슈·결정사항으로 제한한다.
- ❌ 부모가 재사용하지 않을 중간 로그·통계를 반환에 포함하지 않는다.
